# Code Review

## レビュー対象の決定

$ARGUMENTSが指定されている場合はそのファイル・ディレクトリを対象にする。
- ファイルパスが渡された場合 → Read ツールでそのファイルを読む
- ディレクトリパスが渡された場合 → Glob でファイル一覧を取得してから Read で各ファイルを読む

$ARGUMENTSが指定されていない場合は、現在のワーキングディレクトリのコードファイルを対象にする：
1. !`find . -type f \( -name "*.py" -o -name "*.js" -o -name "*.ts" -o -name "*.jsx" -o -name "*.tsx" -o -name "*.vue" -o -name "*.rb" -o -name "*.lua" -o -name "*.bas" -o -name "*.cls" -o -name "*.frm" \) -not -path "*/node_modules/*" -not -path "*/.git/*" -not -path "*/__pycache__/*" -not -path "*/.venv/*" 2>/dev/null | head -30`
2. 上記で見つかったファイルを Read ツールで読む
3. ファイルが多い場合は、ユーザーに「どのファイル・ディレクトリを対象にしますか？」と確認してから進める（目安: 5ファイル超は確認する）

---

## レビュー手順

### Step 1: 言語・技術スタックの判定

対象ファイルの拡張子から使用言語を判定する：
- `.py` → Python
- `.js` / `.ts` / `.jsx` / `.tsx` → JavaScript/TypeScript
- `.vue` → Vue.js
- `.bas` / `.cls` / `.frm` / `.vba` → VBA
- `.lua` → Lua
- `.rb` → Ruby
- `.html` / `.hbs` → HTML/Handlebars
- その他はファイル拡張子から判断

---

### Step 2: 🚨 環境設定チェック【最優先・最初に確認】

**Monitoring モジュール等のURL設定を本番リリース前に必ず確認すること。**
違反があれば他のチェックより先に最優先で警告を出す。

> ⚠️ 本番URLがコメントアウトされている、または検証URLが有効になっている場合は **🚨 クリティカル** として報告する。

---

### Step 3: 共通チェック（全言語）

- [ ] デバッグコードが残っていないか（`print`, `console.log`, `Debug.Print`, `pp` 等）
- [ ] コメントアウトされたままの不要なコードが残っていないか
- [ ] ハードコードされた認証情報・APIキー・パスワードがないか
- [ ] 不要なファイル（`.DS_Store`, `__pycache__`, `*.pyc`, `node_modules` 等）が含まれていないか
- [ ] 変数名・関数名が意味を適切に表しているか（`i`, `r`, `x`, `tmp`, `hoge` 等の1文字・無意味な名前は禁止）

---

### Step 4: 言語固有チェック

#### Python の場合（PEP8 / Google Python Style Guide）

**スタイル**
- [ ] インデントは4スペースか
- [ ] 1行の長さが適切か（PEP8: 79文字、black: 88文字が目安）
- [ ] インポートは stdlib → サードパーティ → ローカルの順に整理されているか（isort）
- [ ] `import *` を使っていないか
- [ ] 不要なインポートがないか

**命名規則**
- [ ] 変数・関数名は `snake_case` か
- [ ] クラス名は `PascalCase` か
- [ ] 定数は `UPPER_SNAKE_CASE` か

**コード品質**
- [ ] 例外処理が適切か（裸の `except:` や `except Exception:` になっていないか）
- [ ] `f-string` や適切な文字列フォーマットを使っているか
- [ ] mutable なデフォルト引数（`def f(x=[]):`）を使っていないか
- [ ] `is None` / `is not None` を使っているか（`== None` は不可）

---

#### JavaScript / TypeScript の場合（ESLint / Prettier）

**スタイル**
- [ ] インデントは2スペースか
- [ ] 文末セミコロンのスタイルが統一されているか

**命名規則**
- [ ] 変数・関数名は `camelCase` か
- [ ] クラス名は `PascalCase` か
- [ ] 定数は `UPPER_SNAKE_CASE` か

**コード品質**
- [ ] `var` を使っていないか（`const` / `let` を使うべき）
- [ ] 非同期処理が適切か（`async/await` を優先）
- [ ] `==` ではなく `===` を使っているか
- [ ] エラーハンドリングが適切か（Promise の `.catch()` や `try/catch`）

**TypeScript 固有**
- [ ] `any` 型を使っていないか（`unknown` を検討）
- [ ] non-null アサーション（`!`）が過剰でないか

**セキュリティ**
- [ ] XSS リスクがないか（`innerHTML`, `dangerouslySetInnerHTML` の不適切な使用）
- [ ] `eval()` を使っていないか

---

#### VBA の場合（Microsoft Office VBA コーディング規約）

**スタイル**
- [ ] インデントは4スペース（またはタブ）で統一されているか
- [ ] 長い行は `_` で適切に折り返されているか

**命名規則**
- [ ] 変数名はハンガリアン記法またはキャメルケースで統一されているか
- [ ] 定数名は `UPPER_SNAKE_CASE` か
- [ ] プロシージャ名・関数名は動詞で始まる `PascalCase` か

**コード品質**
- [ ] `Option Explicit` が全モジュールに宣言されているか
- [ ] 変数の型が明示的に宣言されているか（`Dim x As Long` 等）
- [ ] エラーハンドリング（`On Error GoTo`）が適切か（`On Error Resume Next` の多用は不可）
- [ ] マジックナンバーが使われていないか（`Const` で定数化されているか）
- [ ] ワークシートへの参照が `ActiveSheet` に依存していないか

**エンコーディング**
- [ ] VBA ファイル（`.bas` / `.cls` / `.frm`）内の日本語はSHIFT-JIS（CP932）で出力されているか（UTF-8で書き込むと Windows VBAエディタで文字化けする）

> ⚠️ **VBAファイルを修正・新規作成する場合は必ずSHIFT-JIS（CP932）で書き込むこと。**
> Edit/Write ツールは UTF-8 で書き込むため、VBAファイルには使用禁止。
> 代わりに以下の Bash コマンドで書き込む：
> ```bash
> python3 -c "
> content = '''（VBAコード）'''
> with open('ファイルパス', 'w', encoding='cp932') as f:
>     f.write(content)
> "
> ```

---

#### Lua の場合

**命名規則**
- [ ] 変数・関数名は `snake_case` か
- [ ] 定数は `UPPER_SNAKE_CASE` か

**コード品質**
- [ ] グローバル変数の使用が最小限か（`local` が適切に使われているか）
- [ ] `pcall` / `xpcall` によるエラーハンドリングが適切か
- [ ] `nil` チェックが適切に行われているか

---

#### Ruby の場合

**命名規則**
- [ ] 変数・メソッド名は `snake_case` か
- [ ] クラス・モジュール名は `PascalCase` か

**コード品質**
- [ ] `rescue` で例外クラスが明示されているか（裸の `rescue` は不可）
- [ ] テストが追加・更新されているか

---

### Step 5: Find Bug（バグ検出）

以下の観点でコードを精査し、潜在的なバグを特定する：

**境界値・数値**
- [ ] ゼロ除算のリスクがないか（除算前に分母のゼロチェックがあるか）
- [ ] 配列・コレクションの境界値アクセスが安全か（off-by-one エラー）
- [ ] 数値オーバーフロー・アンダーフローのリスクがないか
- [ ] 浮動小数点の比較に `==` を使っていないか

**null / 未初期化**
- [ ] null / Nothing / None 参照のリスクがないか
- [ ] 変数が初期化される前に参照されていないか
- [ ] 関数の戻り値が null / Nothing になりうる場合に呼び出し元でチェックしているか

**条件分岐・制御フロー**
- [ ] 条件分岐に漏れがないか（`else` / `default` の未処理ケース）
- [ ] 複合条件の論理演算子（`And` / `Or` / `&&` / `||`）が意図通りか
- [ ] `switch` / `Select Case` に `default` / `Case Else` があるか

**ループ**
- [ ] 無限ループになるリスクがないか（終了条件が必ず満たされるか）
- [ ] ループカウンタの更新が適切か
- [ ] コレクションをループ中に変更していないか

**リソース管理**
- [ ] ファイル・DB接続・オブジェクトが確実にクローズ・解放されているか
- [ ] エラー発生時にもリソースが解放されるか（`Finally` / `On Error` での後処理）

**非同期・並行処理**
- [ ] 競合状態（Race Condition）のリスクがないか
- [ ] 非同期処理の結果を待たずに次の処理に進んでいないか

**その他**
- [ ] コピー＆ペーストミス（似たブロックで変数名だけ変え忘れ等）がないか
- [ ] 再帰処理に終了条件があるか（スタックオーバーフローのリスク）

---

### Step 6: スペルチェック

コード内の識別子・コメント・文字列リテラルのスペルミスを検出する：

**識別子（変数名・関数名・定数名・クラス名）**
- [ ] 明らかなタイポがないか（例: `recieve` → `receive`、`caluculate` → `calculate`）
- [ ] 省略形が一貫しているか（例: `msg` と `message` が混在していないか）
- [ ] 単数・複数形が意味に沿っているか（例: コレクションは複数形）

**コメント・文字列リテラル**
- [ ] コメント内の英単語にスペルミスがないか
- [ ] 日本語コメントに誤字・脱字がないか
- [ ] ユーザー向けメッセージ・ログ文字列に誤字がないか

---

### Step 7: チーム固有ルールチェック

#### ルール1: マジックバリューの排除（重要度で区別）

**【重要度：高】ロジックに影響するもの → 厳しく指摘**
- [ ] セルの列番号・行番号・セル位置が直書きされていないか
- [ ] シート名（例: `"Sheet1"`, `"Data"`）が直書きされていないか
- [ ] ファイルパス・ループ制御値が直書きされていないか

**【重要度：低】表示のみのもの → 指摘優先度「低」**
- [ ] メッセージ文言（例: `"処理完了"`, `"エラーです"`）の直書き（指摘はするが必須修正ではない）

#### ルール2: DRY原則（重複コードの排除）
- [ ] 似たような処理が2回以上登場している場合、関数へ切り出されているか
- [ ] コピー＆ペーストで量産されたコードブロックがないか

#### ルール3: モジュールの機能別分割（単一責任の原則）
- [ ] 1つのモジュール・プロシージャに「UI/View操作」「ビジネスロジック」「インフラ/アクセス」が混在していないか
- [ ] 異なる種類の責務が混在している場合、分離・委譲の構成を提案すること

---

### Step 8: ロジック・設計レビュー

- バグやロジックの問題がないか
- 可読性・保守性の観点で改善できる点はないか
- パフォーマンス上の懸念点はないか（N+1クエリ、不要なループ等）
- セキュリティ上のリスクはないか（入力バリデーション、SQLインジェクション等）
- 既存コードとの一貫性が保たれているか
